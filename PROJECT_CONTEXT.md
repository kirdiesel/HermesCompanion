# Исходный контекст проекта TG Companion Bot

## Назначение

`TG Companion Bot` — отдельный Telegram-бот-собеседник, который должен стать главным человеко-понятным интерфейсом взаимодействия Кирилла с Hermes, Obsidian и специализированными агентами.

Бот не должен быть просто ещё одним техническим Telegram-ботом. Его задача — уменьшать трение: принимать задачи обычным сообщением, доводить их до проверяемого результата, выводить результат на приёмку кнопками и фиксировать очищенный итог в Obsidian.

## Главная гипотеза

Один качественный companion-бот с проектными режимами должен сначала закрыть 80% потребностей лучше, чем множество отдельных Telegram-ботов под каждый проект.

Отдельные агенты/боты допустимы позже, когда будет понятно:

- что именно они делают лучше основного companion-бота;
- как они отчитываются;
- как не создают шум;
- как синхронизируются с Obsidian.

## Согласованные UX-правила

### Кнопки

- Кнопки нужны на итоговом результате/приёмке.
- Кнопки не нужны в комментариях, промежуточных статусах, системных сообщениях и простых информационных ответах.
- Количество кнопок зависит от ситуации; использовать минимально достаточное число.
- Тексты кнопок должны быть смысловыми действиями, например:
  - `Принять результат`
  - `Доработать результат`
  - `Показать следующий шаг`
- После подтверждения бот должен выполнять текущую рекомендацию; если рекомендации нет — выбирать следующий оптимальный шаг.

### Статусы

В пользовательском интерфейсе использовать только:

- `▶️ выполняется`
- `🔎 приёмка`
- `✅ готово`

Не использовать как обязательные пользовательские статусы:

- `ждёт`
- `блок`
- `обновлено`

### Приёмка

- Результат не считается готовым до явного подтверждения пользователя.
- До подтверждения задача находится в `🔎 приёмка`.
- После подтверждения задача переходит в `✅ готово`.

### Obsidian

- После каждой принятой задачи обновлять соответствующую проектную заметку.
- В Obsidian попадет очищенный итог, а не сырая переписка.
- По каждому проекту нужен отдельный журнал решений.
- Структура должна быть удобна не только в папках, но и в графе Obsidian.

## Ограничения безопасности

Companion-бот не должен без подтверждения:

- выполнять публичные/внешние действия;
- тратить деньги или менять финансовые данные;
- публиковать контент;
- трогать credentials, токены, auth-файлы;
- массово перемещать/удалять заметки Obsidian;
- менять чувствительные разделы: личное, здоровье, финансы, клиентские данные.

## Текущий фактический статус

На момент создания этого контекста проект является каркасом:

- есть `README.md`;
- нет реализованного кода бота;
- нет выбранного framework/runtime;
- нет `.env` и токена отдельного бота;
- BotFather-бот ещё не создан/не подключён;
- основная работа пока ведётся через текущий Hermes Telegram gateway.

## Ближайшая цель

Подготовить MVP companion-бота так, чтобы можно было безопасно перейти от текущего Hermes Gateway UX к отдельному проектному боту без потери уже согласованных правил взаимодействия.
