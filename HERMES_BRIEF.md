# Бриф Hermes для проекта TG Companion Bot

## Роль проекта

`TG Companion Bot` — пользовательский Telegram-интерфейс для общения с Hermes и агентной системой владельца развёртывания.

Он должен стать точкой входа вида:

> обычное сообщение в Telegram → проверяемый результат → приёмка кнопкой → запись очищенного итога в Obsidian → следующий оптимальный шаг.

## Цель MVP

Сделать минимальный бот/контур, который умеет:

1. принимать задачу из Telegram;
2. определять проект/контекст, если это очевидно;
3. задавать короткое уточнение кнопками, если контекст неоднозначен;
4. передавать задачу Hermes/агентам;
5. возвращать итог с кнопками приёмки;
6. после подтверждения выполнять рекомендацию или следующий оптимальный шаг;
7. сохранять очищенный итог в Obsidian.


## Требование переносимости интерфейса

Созданный интерфейс бота должен быть не одноразовой реализацией только для `tg-companion-bot`, а переносимым стартовым шаблоном для будущих Telegram-ботов с другими подключёнными агентами Hermes One.

Это означает:

- core-логика кнопок, статусов и callback-действий должна жить отдельно от конкретного Telegram framework;
- сценарий `задача → итог → приёмка → Obsidian → следующий шаг` должен переиспользоваться в других ботах;
- labels кнопок и статусы должны конфигурироваться, а не хардкодиться в live adapter;
- проектно-специфичные агенты должны подключаться через слой маршрутизации/адаптеров, а не через копипасту обработчиков;
- новый бот должен стартовать с этого интерфейса как с базового UX-пакета.

## Основные сценарии

### 1. Простая задача

Пользователь пишет задачу обычным сообщением.

Бот должен:

- понять проект/режим;
- выполнить или передать задачу;
- вернуть короткий итог;
- показать кнопки приёмки.

### 2. Неоднозначный контекст

Если непонятно, к какому проекту относится задача, бот должен задать короткий вопрос кнопками, а не гадать.

### 3. Итог задачи

Финальный ответ должен содержать:

- что сделано;
- что проверено;
- текущий статус;
- следующий оптимальный шаг.

Под итогом нужны смысловые кнопки, например:

- `Принять результат`
- `Доработать результат`
- `Показать следующий шаг`

### 4. Подтверждение результата

После `Принять результат` бот должен:

1. снять/обновить кнопки, чтобы они не выглядели активными;
2. перевести задачу в `✅ готово`;
3. обновить Obsidian;
4. выполнить текущую рекомендацию;
5. если рекомендации нет — выбрать следующий оптимальный шаг.

### 5. Доработка

После `Доработать результат` бот должен:

- запросить, что именно изменить, если это не очевидно;
- вернуть задачу в работу;
- не считать её готовой.

## Пользовательские статусы

Использовать только:

- `▶️ выполняется`
- `🔎 приёмка`
- `✅ готово`

## Что не входит в MVP

- отдельные агенты под каждый проект;
- самостоятельные публичные публикации;
- платежи/финансовые действия;
- массовая перестройка Obsidian без приёмки;
- хранение секретов в коде;
- сложная CRM/таск-трекер логика.

## Требования к кнопкам

- Не показывать кнопки под промежуточными комментариями.
- Не показывать кнопки под системными сообщениями.
- Не показывать кнопки под простыми информационными ответами.
- Показывать кнопки только там, где они ускоряют действие или нужны для приёмки/выбора.

## Требования к Obsidian

После принятой задачи обновлять:

- проектную заметку;
- журнал решений проекта;
- при необходимости — список следующих шагов.

Писать только очищенный итог:

- дата;
- задача;
- решение;
- проверка;
- следующий шаг;
- ссылки на артефакты.

## Критерии готовности MVP

MVP считается готовым, если:

1. бот запускается локально без ошибок;
2. принимает Telegram-сообщение;
3. возвращает итоговый ответ;
4. показывает кнопки только на итоговом ответе;
5. корректно обрабатывает `Принять результат` и `Доработать результат`;
6. обновляет Obsidian после принятия;
7. имеет тесты на рендер кнопок/статусов;
8. не конфликтует с текущим Hermes Gateway polling.

## Риски

- конфликт long polling, если бот использует тот же token, что Hermes Gateway;
- избыточные кнопки и Telegram-шум;
- попадание сырой переписки в Obsidian;
- попытка сделать слишком сложную архитектуру до MVP;
- смешивание личного, рабочего и клиентского контекста.

## Рекомендованная стратегия реализации

1. Сначала описать архитектуру и сценарии.
2. Затем выбрать framework.
3. Затем сделать renderer сообщений и кнопок с тестами.
4. Затем добавить Telegram handlers.
5. Затем добавить Obsidian persistence.
6. Только после этого подключать реальный token и live-тест.
